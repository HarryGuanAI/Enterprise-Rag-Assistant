# Enterprise RAG Assistant 项目复盘与面试准备

日期：2026-05-12  
项目名称：Enterprise RAG Assistant / 云舟知识库助手  
求职方向：AI Agent 开发、RAG 应用开发、LLM 应用工程师  
项目目标：做出一个可部署、可演示、可开源，并且能在面试中讲清楚工程链路和技术取舍的企业知识库 RAG 助手。

## 1. 项目起点

这个项目是为了 AI Agent 开发岗位求职准备的第一个作品。最初目标不是做一个“代码练习”，而是做一个能放到 GitHub、能部署、能打开网页展示、能讲清楚架构和优化思路的真实项目。

项目定位最终确定为：

```text
企业内部知识库 RAG 助手
```

示例业务背景是虚构企业“云舟科技 Yunzhou Cloud”，场景是企业员工围绕制度、报销、员工手册、IT 指南、产品 FAQ 等内部文档提问，系统基于知识库检索结果回答，并展示引用来源。

这个选题适合 AI Agent / RAG 岗位，因为它覆盖了：

- 文档上传与解析
- 文档分块
- Embedding
- 向量数据库
- 语义检索
- 严格拒答
- LLM 流式生成
- 引用来源
- 管理员权限
- 成本控制
- 基础评测
- 后续 Agent 化扩展

## 2. 需求讨论过程与最终决策

### 2.1 第一版做什么

第一版目标确定为“可用、可部署、可展示”的成品，而不是只写一个最小 Demo。

最终第一版范围：

- 单页网页工作台
- 管理员登录
- 游客模式
- 游客只能问 2 个问题
- 管理员可以上传文档
- 支持文档解析、分块、入库
- 支持真实 Embedding
- 支持真实 LLM 问答
- 支持引用来源
- 支持严格拒答
- 支持基础统计
- 支持 RAG 参数设置
- 支持基础评测
- 支持 Docker Compose 部署

暂不做：

- 多用户注册
- 完整 RBAC 权限系统
- 多知识库管理页面
- OCR
- `.doc` 老格式解析
- 复杂表格还原
- LangChain 主链路
- LangGraph Agent 化流程
- 生产级异步任务队列

面试讲法：

> 我第一版先控制边界，优先做完整 RAG 主链路和可演示体验。多用户、OCR、Agent 编排这些放到 Roadmap，因为它们会增加复杂度，但不是验证 RAG 核心能力的必要条件。

### 2.2 为什么选企业知识库 RAG

我们讨论过，AI Agent 开发岗位并不只看“会调用大模型 API”，更看能不能把大模型接到真实业务流程里。

企业知识库 RAG 是一个很好的求职项目，因为它贴近实际公司场景：

- 公司内部文档多
- 员工经常查制度、流程、FAQ
- 直接让模型自由回答容易幻觉
- RAG 可以把回答限制在企业资料范围内
- 引用来源能增强可信度
- 严格拒答可以降低错误回答风险

面试讲法：

> 这个项目不是简单聊天机器人，而是一个有知识边界的企业问答系统。核心目标是让模型“基于资料回答”，并且在资料不足时明确拒答，而不是编造答案。

### 2.3 前端是不是网页，后端是不是部署到服务器的程序

讨论中明确过：

- 前端就是用户能打开的网页
- 后端是部署到服务器或容器里的 API 服务
- 前端负责交互、展示、上传、问答、引用来源
- 后端负责认证、文档处理、检索、调用模型、保存数据

面试讲法：

> 前端是 Next.js 单页工作台，后端是 FastAPI 服务。前端不直接接触 API Key，只通过后端接口完成上传、检索和问答，保证密钥和核心逻辑在服务端。

### 2.4 为什么要做管理员登录和游客限制

用户提出需要内置登录，并且游客模式只能回答两个问题，原因是担心公开网页后有人恶意消耗 API Token。

最终实现方向：

- 内置管理员账号
- 管理员可以上传文档和修改设置
- 游客可以体验问答
- 游客每日 2 次限制
- 后端做限制校验
- 增加 IP 级兜底限制

面试讲法：

> 这是成本控制和安全边界设计。因为 RAG 系统会调用 Embedding 和 LLM，如果演示页面公开，必须限制游客调用次数，不能只在前端限制，必须在后端校验。

### 2.5 为什么第一版不支持 `.doc`

讨论过为什么不支持 `.doc` 文档解析。

原因：

- `.doc` 是旧版二进制格式，不像 `.docx` 那样是结构化压缩包
- 解析稳定性差
- 通常需要 LibreOffice、antiword 或系统级转换工具
- Docker 部署会变重
- 第一版会增加太多非核心复杂度

最终决策：

- 第一版支持 `.docx`
- 第一版不支持 `.doc`
- Roadmap 里可以通过 LibreOffice 先转 `.docx` 或 PDF 再解析

面试讲法：

> 我没有盲目支持所有格式，而是明确第一版边界。`.doc` 需要额外系统依赖，容易影响部署稳定性，所以先支持 PDF、DOCX、Markdown、TXT，把 RAG 主链路做稳。

### 2.6 为什么先不用 LangChain

用户问过：以后肯定要用 LangChain，那什么时候用？

最终思路：

- 第一版先不用 LangChain，自己实现轻量 RAG 链路
- 目的是理解底层流程，而不是一上来被框架包装
- 后续项目或第二阶段可以做 LangChain 对照版
- LangChain 适合复杂链路、工具调用、Retriever 组合、Agent 编排
- 当前第一版更适合手写主链路，便于面试讲清楚细节

面试讲法：

> 我第一版没有直接用 LangChain，是为了把 RAG 的关键环节拆开实现：解析、分块、Embedding、向量检索、阈值拒答、Prompt 拼接、流式生成。这样我能讲清楚每一步。后续会补一个 LangChain 或 LangGraph 版本，展示框架化和 Agent 化能力。

## 3. 技术选型复盘

### 3.1 总体技术栈

前端：

- Next.js 14
- React 18
- Tailwind CSS
- TanStack Query
- lucide-react
- SSE 流式接收

后端：

- FastAPI
- SQLAlchemy 2.x
- Pydantic
- Alembic
- PostgreSQL
- pgvector
- Docker Compose

模型：

- DashScope `text-embedding-v4`
- DeepSeek `deepseek-chat`

### 3.2 为什么选 Next.js

原因：

- 适合快速构建完整网页应用
- React 生态成熟
- 和 Tailwind 搭配开发效率高
- 容易做单页工作台
- 后续开源和部署都比较常见

第一版实际使用：

- 单页工作台
- 顶部统计区
- 左侧文档列表
- 中间聊天区
- 右侧引用来源
- 登录弹窗
- 设置弹窗

面试讲法：

> 前端选择 Next.js 是为了快速做出一个真实可用的产品界面，而不是只做接口。RAG 项目最终要能演示，前端体验会影响面试官对工程完整度的判断。

### 3.3 为什么选 FastAPI

原因：

- Python 生态适合 AI 应用开发
- FastAPI 对异步、SSE、Pydantic 支持好
- 接口定义清晰
- 和文档解析、模型调用库配合方便

面试讲法：

> FastAPI 很适合 LLM 应用后端。它可以同时处理普通 REST 接口和 SSE 流式输出，Pydantic 也方便做请求响应校验。

### 3.4 为什么选 PostgreSQL + pgvector

原因：

- PostgreSQL 同时存业务数据和向量数据
- pgvector 可以做向量检索
- 第一版不用额外引入 Milvus、Qdrant、Elasticsearch，部署更简单
- 适合中小型 RAG 项目
- 后续仍可扩展 Hybrid Search

面试讲法：

> 我选择 PostgreSQL + pgvector 是为了降低系统复杂度。业务数据、文档元数据、消息记录、向量都可以放在同一个数据库里。第一版更容易部署和讲清楚，后续如果数据量变大，可以迁移到独立向量数据库或搜索引擎。

### 3.5 为什么用 DashScope Embedding

用户已有阿里云百炼 / DashScope Key，所以第一版选择 DashScope。

当前使用：

- 模型：`text-embedding-v4`
- 维度：1024
- 写入 pgvector `vector(1024)`

面试讲法：

> Embedding 模型负责把文本变成向量。这里我用 DashScope text-embedding-v4，生成 1024 维向量，再用 pgvector 做相似度检索。

### 3.6 为什么用 DeepSeek

用户已有 DeepSeek API，所以生成模型先用 DeepSeek。

当前使用：

- `deepseek-chat`
- Chat Completions 兼容接口
- SSE 流式输出

面试讲法：

> DeepSeek 负责最终回答生成，但它不是自由发挥，而是基于检索出来的上下文回答。上下文不足时系统不调用 DeepSeek，直接严格拒答，以降低幻觉和 API 成本。

## 4. 架构复盘

### 4.1 整体架构

```text
Browser
  |
  | HTTP / SSE
  v
Next.js Frontend
  |
  | REST API
  v
FastAPI Backend
  |
  | SQL + Vector Search
  v
PostgreSQL + pgvector

FastAPI Backend
  |                         |
  | Embedding API            | Chat Completion API
  v                         v
DashScope                   DeepSeek
```

### 4.2 后端分层

后端按职责拆分：

- `api/routes`：接口层
- `services`：业务服务层
- `rag/loaders`：文档解析
- `rag/splitters`：文档分块
- `rag/embeddings`：Embedding 适配器
- `rag/retrievers`：检索器
- `rag/generators`：生成模型适配器
- `models`：数据库模型
- `schemas`：Pydantic 模型
- `evals`：评测脚本

面试讲法：

> 我没有把 RAG 逻辑全写在一个接口里，而是拆成文档服务、检索器、Embedding 适配器、生成器、设置服务等模块。这样后续替换模型、替换检索策略、加入 Rerank 都比较清晰。

### 4.3 前端结构

前端核心文件：

- `frontend/app/page.tsx`：主工作台
- `frontend/lib/api.ts`：API 与 SSE 封装
- `frontend/types/api.ts`：接口类型

当前页面结构：

- 顶部：产品名、身份状态、统计卡片
- 左侧：文档列表、上传、重新入库
- 中间：聊天区、答案区、输入区
- 右侧：引用来源、检索调试
- 弹窗：登录、RAG 设置

最近优化：

- 页面改成固定视口高度
- 输入框固定在聊天面板底部
- 左中右各自滚动
- 长引用不会再把整页撑得很长

面试讲法：

> RAG 系统不只是后端算法，前端也要让用户看到答案依据。我做了引用来源和检索调试，让系统的回答过程更可解释。

## 5. 数据模型复盘

核心表：

| 表 | 作用 |
|---|---|
| `knowledge_bases` | 知识库表，第一版只有默认知识库 |
| `documents` | 文档表，保存上传文件和处理状态 |
| `document_chunks` | 文档分块表，保存文本、元数据和 embedding |
| `conversations` | 会话表 |
| `messages` | 消息表 |
| `message_citations` | 结构化引用表 |
| `guest_usage` | 游客使用次数表 |
| `app_settings` | RAG 参数设置表 |
| `model_call_logs` | 模型调用记录表 |

关键设计：

- 核心业务表主键使用 UUID
- `documents` 记录处理状态：`processing`、`ready`、`failed`
- `document_chunks` 保存 chunk 文本、章节路径、向量
- 引用来源既保存 JSON 快照，也保存结构化关联

为什么引用要“双存储”：

- `messages.citations_json` 保存当次回答展示快照
- `message_citations` 保存 message 和 chunk 的结构化关联

面试讲法：

> 引用来源我做了快照和结构化关联两种保存。快照保证历史回答稳定，结构化关联方便后续统计哪些 chunk 经常被引用、哪些问题容易失败。

## 6. RAG 主链路复盘

### 6.1 文档入库链路

```text
上传文档
-> 校验格式和大小
-> 保存原始文件
-> 创建 documents 记录
-> 状态标记 processing
-> 后台解析文档
-> 混合分块
-> 调用 DashScope Embedding
-> 写入 document_chunks 和 pgvector
-> 状态标记 ready
```

当前支持格式：

- PDF
- DOCX
- Markdown
- TXT

不支持：

- `.doc`
- 扫描版 PDF OCR
- 复杂表格结构还原

### 6.2 问答链路

```text
用户提问
-> 身份和游客次数校验
-> 生成问题向量
-> pgvector Top-K 检索
-> 计算 Top1 相似度
-> 低于阈值则严格拒答
-> 命中可靠上下文则拼接 Prompt
-> 调用 DeepSeek 流式生成
-> 返回答案增量
-> 返回引用来源
-> 保存消息和引用
```

### 6.3 严格拒答

当前默认：

- `top_k = 5`
- `min_similarity = 0.60`
- `strict_refusal = true`

为什么从 `0.35` 调到 `0.60`：

- 最初阈值 `0.35` 太宽
- 评测发现知识库外问题也可能命中一些弱相关内容
- 这会导致系统错误回答
- 调整到 `0.60` 后，10 条基准样例中应拒答问题全部正确拒答

面试讲法：

> RAG 不只是召回，还要判断“能不能答”。我用 Top1 相似度做严格拒答阈值，低于阈值直接拒答，并且不调用 LLM。这样既降低幻觉，也节省模型调用成本。

### 6.4 SSE 流式输出

当前 SSE 事件：

- `status`：当前处理阶段
- `retrieval_debug`：检索调试信息
- `citations`：引用来源
- `answer_delta`：答案增量
- `done`：完成
- `error`：错误

为什么不用一次性 HTTP 返回：

- 流式回答体验更接近真实 AI 产品
- 用户可以看到生成过程
- 后端可以分阶段返回检索信息和引用来源
- 错误处理更清晰

面试讲法：

> 我用 SSE 做流式输出，而不是 WebSocket，因为这个场景是服务端单向持续推送，SSE 更简单、稳定，也更适合第一版 RAG 问答。

## 7. 前端体验复盘

### 7.1 早期问题

开发过程中遇到过：

- 访问 `localhost:3000` 连接被拒绝
- `127.0.0.1:3000` 仍然打不开
- 后来确认是前端服务没有在对应端口启动
- 多次换端口，最终当前前端稳定在 `3004`

面试讲法：

> 本地开发时我区分了服务是否启动、端口是否占用、前端是否能访问后端 API。最终把前端固定在 3004，后端固定在 8000。

### 7.2 上传和发送按钮无反应

早期前端还只是静态骨架，所以点击上传和发送没有真实逻辑。后来逐步接入：

- 登录态
- 上传接口
- 文件选择
- multipart 上传
- SSE 问答接口
- 流式回答展示
- 引用来源展示

### 7.3 管理员密码问题

用户要求调整管理员密码。之后出现过登录失败，原因是前端/后端服务和环境变量需要重启同步。

最终：

- `.env` 中 `ADMIN_PASSWORD` 设置为本地私有值
- 后端读取环境变量
- 管理员可登录

面试讲法：

> 认证配置放在环境变量里，不硬编码到代码中。修改后需要重启后端服务，让新配置生效。

### 7.4 回答 Markdown 显示问题

用户指出模型回答里出现 `**标题**` 这类 Markdown 符号，页面显示凌乱。

我们处理了：

- `**重点**` 显示为加粗
- 短横线列表结构化显示
- 数字列表结构化显示
- 回答气泡样式优化

解释：

- 双星号在 Markdown 里表示加粗
- 模型常用它强调标题或重点
- 前端应该渲染它，而不是直接显示原始符号

面试讲法：

> LLM 输出经常带 Markdown，因此前端需要做基础 Markdown 渲染。否则答案内容是对的，但展示体验会很差。

### 7.5 页面太长问题

用户指出页面太长，每次输入都要往下拉。

优化后：

- 主页面固定为 `h-screen`
- 整页不再长滚动
- 左侧文档列表内部滚动
- 中间聊天记录内部滚动
- 右侧引用来源内部滚动
- 输入区固定在聊天面板底部
- 新消息自动滚动到底部

面试讲法：

> RAG 问答会产生长答案和长引用，如果让整页滚动，输入体验会很差。我把页面改成固定工作台布局，让聊天、文档、引用各自滚动。

## 8. 评测复盘

### 8.1 为什么要做评测

RAG 优化不能只靠主观感觉，需要有基准问题。

第一版评测目标：

- 检索是否命中正确文档
- 应该拒答的问题是否拒答
- 答案是否覆盖关键点

当前文件：

- `evals/golden_qa.jsonl`
- `backend/app/evals/run_eval.py`

### 8.2 当前评测集

当前包含 10 条样例：

- 员工手册相关问题
- 报销制度相关问题
- 应该拒答的问题

示例问题：

- 新员工入职第一周需要完成哪些事项？
- 出差报销需要提交哪些材料？
- 公司年会抽奖一等奖是什么？
- 员工股票期权怎么归属？
- 公司食堂早餐几点开始？

### 8.3 评测结果

调阈值前：

- 检索命中率：1.00
- 拒答准确率：0.70
- 问题：知识库外问题被弱相关内容放过去

调阈值后：

- 检索命中率：1.00
- 拒答准确率：1.00
- 关键词覆盖均值：1.00

面试讲法：

> 我不是凭感觉调阈值，而是用小型 golden set 做验证。评测发现 0.35 太宽，所以调到 0.60，拒答准确率从 0.70 提升到 1.00。

## 9. 安全与成本控制复盘

已经考虑的点：

- API Key 只放后端 `.env`
- `.env` 加入 `.gitignore`
- 游客不能上传文档
- 游客问答次数限制
- IP 日限制
- 低相关问题不调用 DeepSeek
- 上传文件格式和大小限制
- 管理员接口鉴权

重要安全提醒：

- 用户曾在对话中暴露过真实 API Key

## 17. 检索优化复盘：Hybrid Search 与轻量 Rerank

这一轮已经把原来设置面板里的 `Hybrid Search` 和 `Rerank` 占位开关接入真实检索链路。它是当前项目最适合在面试里展开讲的优化点之一，因为它能体现对 RAG 系统“召回、排序、拒答、成本、评测”的整体理解。

### 17.1 为什么要做检索优化

第一版纯向量检索已经能跑通完整 RAG 主链路，但真实企业知识库里会遇到一些纯向量检索不稳定的场景：

- 专有名词、制度名称、缩写、编号、条款原文需要精确匹配。
- 用户问题可能很短，语义信息不足，Embedding 召回容易飘。
- 多个文档内容语义接近时，仅靠向量分不一定能把最准确的制度片段排到最前。
- 严格拒答依赖 Top1 分数，如果检索分数校准不稳，就可能误答或误拒答。

所以这一轮没有直接堆一个复杂模型，而是先把检索过程拆成两层：

```text
初召回：向量召回 + 关键词召回
二次排序：轻量 Rerank
```

面试讲法：

> 纯向量检索适合语义泛化，但对制度编号、专有名词、原文短语不一定稳定。所以我加入 Hybrid Search，用关键词召回补足精确匹配能力，再通过轻量 Rerank 调整排序。这样既提升检索可解释性，也不会立刻增加额外模型成本。

### 17.2 当前 Hybrid Search 怎么实现

开启 `enable_hybrid_search` 后，检索器会同时做两路召回：

1. 向量召回
   - 使用 DashScope `text-embedding-v4` 生成问题向量。
   - 使用 PostgreSQL + pgvector 的 cosine distance 检索候选 chunks。
   - 适合同义问法、自然语言改写和语义相近问题。

2. 关键词召回
   - 从中文问题中抽取连续短语、4-gram、3-gram、2-gram。
   - 过滤一部分常见虚词。
   - 在文档标题、章节路径、chunk 内容中做匹配。
   - 根据标题/章节命中、内容命中、重复命中、整句命中计算关键词分。

两路结果会按 chunk 去重后融合排序。向量和关键词都命中的片段会得到更高置信度；只有关键词命中的片段也可以补召回，但分数上限会更保守，避免轻易绕过严格拒答。

面试讲法：

> 我把 Hybrid Search 做成向量召回和关键词召回的融合。向量负责语义相似，关键词负责精确词匹配，尤其适合企业制度里的专有名词和条款。融合时我不会让关键词独有命中过度抬高分数，因为 RAG 系统还要兼顾拒答安全。

### 17.3 当前轻量 Rerank 怎么实现

开启 `enable_rerank` 后，系统会对融合后的候选做一次轻量精排：

- 不额外调用大模型或 rerank 模型。
- 使用向量分、关键词覆盖、标题/章节命中共同计算排序分。
- Rerank 只负责调排序和小幅加权，不降低已有向量召回的置信度。

这一点很重要。第一次评测时发现，轻量 Rerank 如果把分数重新校准得太保守，会把一个本来应该回答的问题压到 `min_similarity=0.60` 以下，导致误拒答。因此最终策略是：Rerank 可以提高更相关片段的排序，但不能破坏原本已经通过验证的严格拒答阈值。

面试讲法：

> Rerank 的目标是提高排序质量，而不是重新发明一套不可控的置信度体系。所以我让它不降低已有向量分，避免把原来能回答的问题误拒答。这个细节体现了 RAG 优化不能只看排序，还要看它对拒答策略的影响。

### 17.4 前端如何展示这个优化

右侧“检索调试”现在会展示：

- 当前检索模式：`vector`、`hybrid`、`vector+rerank`、`hybrid+rerank`
- 向量候选数量
- 关键词候选数量
- 最终入选数量
- 每条引用的总分、向量分、关键词分和来源类型

这让面试演示时可以直接解释系统为什么命中某个片段，而不是只展示一个黑盒答案。

面试讲法：

> 我在前端保留了检索调试信息。面试或排查问题时，可以看到本次是纯向量还是 Hybrid+Rerank，召回了多少候选，每个引用片段的向量分和关键词分是多少。这让 RAG 链路更可解释。

- Hybrid + Rerank 模式：检索命中率 1.00，拒答准确率 1.00。

### 17.6 后续可以怎么升级
