# 企业知识库 RAG 助手需求说明

## 1. 项目定位

Enterprise RAG Assistant 是一个面向企业内部员工的知识库问答系统。页面产品名为“云舟知识库助手”。员工可以围绕企业制度、IT 指南、产品 FAQ、销售客服话术等资料提问，系统基于知识库检索结果生成答案，并展示引用来源。

项目服务两个目标：

1. 做成可部署、可演示的开源成品。
2. 帮助求职者在 AI Agent / RAG 开发面试中讲清楚完整工程链路。

## 2. 示例业务背景

虚构公司：云舟科技 Yunzhou Cloud  
公司类型：企业协作 SaaS 公司

示例知识库包含：

- 员工手册
- 请假与考勤制度
- 报销制度
- IT 服务指南
- 产品 FAQ
- 销售与客服话术

第一版页面只展示一个默认企业知识库，避免功能过度复杂。数据库和后端模型预留 `knowledge_base_id` 字段，为后续扩展 HR 知识库、IT 知识库、产品知识库等多知识库管理做准备。

## 3. 用户角色

### 游客

- 可以直接进入页面体验问答。
- 只能基于已有示例知识库提问。
- 每个游客会话每天最多 15 次问答。
- 不能上传、删除、重建或清空知识库。

### 管理员

- 使用内置账号密码登录。
- 可以上传文档。
- 可以删除文档。
- 可以清空知识库。
- 可以重建索引。
- 可以调整 RAG 参数。
- 可以不受游客问答次数限制使用问答。

## 4. 第一版功能范围

### 4.1 单页工作台

页面分区：

- 顶部：项目名、模型状态、统计卡片、设置入口、登录状态。
- 左侧：文档上传、文档列表、文档状态。
- 中间：聊天问答区，支持 SSE 流式输出。
- 右侧：引用来源列表，展示文档名、章节路径、片段内容和相关性分数。

### 4.2 文档管理

支持格式：

- `.pdf`
- `.docx`
- `.md`
- `.txt`

第一版边界：

- PDF 支持文本型 PDF。
- 不支持扫描件 OCR。
- 不支持老 `.doc`。
- 不做复杂表格结构还原。

文档状态：

- `processing`：处理中
- `ready`：已入库
- `failed`：处理失败

### 4.3 RAG 问答

核心流程：

1. 用户输入问题。
2. 后端校验身份和配额。
3. 使用 DashScope Embedding 生成问题向量。
4. 使用 pgvector 检索相关文档片段。
5. 可选启用 Hybrid Search。
6. 可选启用 Rerank。
7. 低相关时严格拒答。
8. 有可靠上下文时调用 DeepSeek 流式生成答案。
9. 保存聊天历史和引用来源。
10. 前端展示回答和引用。

### 4.4 设置面板

管理员可配置：

| 分类 | 参数 | 默认值 | 是否需要重建索引 |
|---|---:|---:|---|
| 检索 | TopK | 5 | 否 |
| 检索 | 相似度阈值 | 0.35 | 否 |
| 检索 | 严格拒答 | 开启 | 否 |
| 检索 | Hybrid Search | 关闭 | 否 |
| 检索 | Rerank | 关闭 | 否 |
| 分块 | Chunk Size | 800 | 是 |
| 分块 | Chunk Overlap | 120 | 是 |
| 生成 | Temperature | 0.2 | 否 |
| 生成 | Max Tokens | 1200 | 否 |

### 4.5 基础统计

展示：

- 文档总数
- 已入库文档数
- 分块总数
- 今日问答次数
- 今日拒答次数
- 游客剩余次数
- 今日模型调用次数
- 今日 Embedding 任务次数

第一版只展示调用次数和问答次数，不估算金额。模型价格和接口返回字段可能变化，成本估算放入后续 Roadmap。

### 4.6 基础评测

第一版包含基础 RAG 评测集和脚本。

评测数据包含：

- 用户问题
- 期望来源文档
- 期望答案关键词
- 问题分类

评测指标：

- `source_hit_rate`：期望来源是否出现在 TopK 检索结果中。
- `keyword_coverage`：最终答案是否覆盖期望关键词。
- `refusal_count`：被严格拒答的样本数量。

第一版不接入大模型自动评分，避免额外成本和评分不稳定。后续可加入 LLM-as-a-Judge 作为增强能力。

## 5. 非功能需求

### 5.1 成本控制

- API Key 只保存在后端环境变量中。
- 游客不能上传文档，避免触发 Embedding 成本。
- 游客问答限制必须在后端校验。
- 低相关问题拒答时不调用 DeepSeek。

### 5.2 安全边界

- 限制上传文件类型和大小。
- 清洗文件名，防止路径穿越。
- 管理员接口必须鉴权。
- 危险操作如清空知识库需要二次确认。

### 5.3 可读性

- 代码结构按 API、Service、RAG 模块、模型适配器分层。
- 关键业务逻辑提供必要中文注释。
- README 和 docs 解释技术取舍、部署步骤和面试讲法。

## 6. Roadmap

第二阶段：

- LangChain 对照实现。
- Query Rewrite。
- 父子分块。
- 更完整的 Rerank 指标。
- Elasticsearch/OpenSearch Hybrid Search。

第三阶段：

- 多知识库。
- 多用户和 RBAC。
- Celery + Redis 文档处理 worker。
- OCR。
- `.doc` 转换解析。
- LangGraph Agent 化，将 RAG 封装为企业流程 Agent 的 Tool。
